# 岗位筛选规则

本文件定义了定时搜索任务中岗位筛选的完整流程。Claude 在环节 7 读取此文件，对搜索到的岗位逐条执行筛选。

## 输入来源

| 文件 | 用途 |
|------|------|
| `data/job-profile.md` | 用户求职画像（城市、薪资、休息制度、公司偏好等） |
| `data/meta.json` | 运行元数据（`recorded_job_ids`、`exclude_keywords`） |
| `data/job-excluded.csv` | 已排除岗位的 `security_id` 列表 |

---

## 筛选流程

按以下步骤顺序执行。任一步骤判定不通过，该岗位立即排除，不再执行后续步骤。

### Step 1: ID 去重

检查岗位的 `security_id` 是否存在于 `meta.json.recorded_job_ids` 数组中。

- 已存在 → 排除，原因标记 `DUPLICATE`
- 不存在 → 通过，继续下一步

### Step 1.5: 排除列表检查

读取 `data/job-excluded.csv` 中所有 `security_id`。

- 岗位的 `security_id` 在排除列表中 → 排除，原因标记 `EXCLUDED`
- 不在排除列表中 → 通过，继续下一步

如果 `job-excluded.csv` 不存在或为空（仅表头行），跳过此步骤。

### Step 2: 排除关键词

检查岗位名称（`job_name`）和岗位描述（`job_description`）是否包含 `meta.json.exclude_keywords` 中的任一关键词。匹配不区分大小写。

- 命中任一关键词 → 排除，原因标记 `EXCLUDE_KEYWORD:<命中的关键词>`
- 未命中 → 通过，继续下一步

当 `exclude_keywords` 为空数组时，跳过此步骤。

### Step 3: 城市区域

读取 `job-profile.md` 中的城市信息：
- `城市` — 目标城市
- `优先区域` — 优先考虑的区域列表
- `排除区域` — 明确排除的区域列表

匹配规则：

1. **精确匹配优先**：岗位地点包含优先区域中的任一区域名称 → 通过
2. **宽松匹配**：岗位地点包含城市名称，且不包含任何排除区域 → 通过
3. **远程岗位**：岗位地点明确标注"支持远程"或"远程办公" → 通过，附加标记 `REMOTE`
4. 以上均不满足 → 排除，原因标记 `LOCATION_MISMATCH`

当 `排除区域` 为空或标注"无"时，不执行排除区域检查。

### Step 4: 薪资下限

读取 `job-profile.md` 中 `薪资期望` 的 `最低` 值作为阈值。

解析岗位薪资字段，规则如下：

| 薪资格式 | 解析方式 |
|---------|---------|
| `15-25K` | 最低 15000，最高 25000 |
| `15-25K·13薪` | 最低 15000，最高 25000，附加标记 `HAS_YEAR_BONUS:13` |
| `15-25K·14薪` | 最低 15000，最高 25000，附加标记 `HAS_YEAR_BONUS:14` |
| `15-25K·15薪` | 最低 15000，最高 25000，附加标记 `HAS_YEAR_BONUS:15` |
| `面议` | 跳过薪资筛选，直接通过 |
| `日薪 XXX` / `时薪 XXX` | 按每月 22 个工作日 / 每天 8 小时换算为月薪后比较 |

判定逻辑：
- 岗位薪资下限 >= 用户最低薪资 → 通过
- 岗位薪资下限 < 用户最低薪资 → 排除，原因标记 `SALARY_BELOW_MIN:<岗位下限>_<用户最低>`
- 薪资字段为"面议" → 通过，附加标记 `SALARY_NEGOTIABLE`

### Step 5: 休息制度

读取 `job-profile.md` 中 `休息制度` 的 `最低` 值。休息制度从严到宽排列：

```
单休 < 单双休 < 双休
```

判定逻辑：

- 岗位明确标注"双休" → 通过
- 岗位明确标注"单双休" → 若用户最低要求为"单双休"或"单休"则通过，否则排除
- 岗位明确标注"单休" → 若用户最低要求为"单休"则通过，否则排除，原因标记 `REST_DAY:单休`
- 岗位未标注休息制度 → 通过（不因缺失信息排除）

### Step 6: HR 活跃度

检查岗位详情中的 `active_time` 字段（HR 活跃时间）。

活跃度等级从高到低：

```
刚刚活跃 > 今日活跃 > 3日内活跃 > 本周活跃 > 2周内活跃 > 更低频/无活跃
```

判定逻辑：

- "刚刚活跃" / "今日活跃" / "3日内活跃" / "本周活跃" → 通过
- "2周内活跃" / "半个月内活跃" / "1个月内活跃" / 更低频 → 排除，原因标记 `HR_INACTIVE:<active_time>`
- `active_time` 字段缺失 → 通过（不因缺失信息排除），附加标记 `HR_ACTIVITY_UNKNOWN`

### Step 7: 公司规模（软条件）

读取 `job-profile.md` 中 `公司偏好` 的 `最低规模` 值。

判定逻辑：

- 公司规模 >= 最低规模 → 通过
- 公司规模 < 最低规模 → 通过，但附加标记 `SCALE_PENALTY`
- 公司规模信息缺失 → 通过，不附加标记

**注意**：公司规模是软条件，不排除岗位，仅影响后续评分环节的得分。附加 `SCALE_PENALTY` 标记的岗位在评分时公司规模维度得 0 分。

## 薪资解析规则汇总

```
输入格式                    最低薪资    最高薪资    附加标记
─────────────────────────────────────────────────────────────
"15-25K"                   15000       25000       无
"15-25K·13薪"              15000       25000       HAS_YEAR_BONUS:13
"15-25K·14薪"              15000       25000       HAS_YEAR_BONUS:14
"15-25K·15薪"              15000       25000       HAS_YEAR_BONUS:15
"面议"                      —           —           SALARY_NEGOTIABLE
"300元/天"                  6600        6600        无（按22天/月换算）
```

解析时注意：
- K 等价于 1000
- 薪资范围中可能出现小数，如 `12.5-18K` → 最低 12500
- 若薪资格式无法识别，跳过薪资筛选，附加标记 `SALARY_UNPARSABLE`

## 地点匹配规则汇总

```
匹配级别        条件                                    结果
─────────────────────────────────────────────────────────────────
精确匹配        岗位地点包含优先区域名称                  通过
宽松匹配        岗位地点包含城市名 且 不在排除区域         通过
远程豁免        岗位标注"远程"相关字样                    通过（标记 REMOTE）
不匹配          以上均不满足                              排除
```

## 输出格式

筛选完成后，生成以下两部分结果：

### 通过筛选的岗位

```
## 通过筛选

| # | 岗位名称 | 公司 | 薪资 | 地点 | 发布日期 | 标记 |
|---|---------|------|------|------|---------|------|
| 1 | xxx     | xxx  | xxx  | xxx  | xxx     | REMOTE, HAS_YEAR_BONUS:14 |
```

标记字段可能包含的值：
- `REMOTE` — 远程岗位
- `SALARY_NEGOTIABLE` — 面议薪资
- `SALARY_UNPARSABLE` — 薪资无法解析
- `HAS_YEAR_BONUS:N` — N 个月年终奖
- `HR_ACTIVITY_UNKNOWN` — HR 活跃度信息缺失
- `SCALE_PENALTY` — 公司规模低于最低要求

### 被排除的岗位

```
## 被排除

| # | 岗位名称 | 公司 | 排除原因 |
|---|---------|------|---------|
| 1 | xxx     | xxx  | DUPLICATE |
| 2 | xxx     | xxx  | EXCLUDE_KEYWORD:实习 |
| 3 | xxx     | xxx  | LOCATION_MISMATCH |
| 4 | xxx     | xxx  | SALARY_BELOW_MIN:10000_15000 |
| 5 | xxx     | xxx  | REST_DAY:单休 |
```

排除原因枚举值：
- `DUPLICATE` — 已记录过的岗位
- `EXCLUDED` — 用户人工排除的岗位
- `EXCLUDE_KEYWORD:<关键词>` — 命中排除关键词
- `LOCATION_MISMATCH` — 地点不匹配
- `SALARY_BELOW_MIN:<岗位下限>_<用户最低>` — 薪资低于下限
- `REST_DAY:<休息制度>` — 休息制度不符合要求
- `HR_INACTIVE:<活跃时间>` — HR 活跃度过低

## 执行注意事项

1. 筛选按 Step 1 到 Step 7 顺序执行，短路求值：任一步骤排除后不再执行后续步骤
2. 所有标记字段原样传递到后续评分环节（`prompts/scorer.md`），不要丢弃
3. 被排除的岗位不计入 `meta.json.recorded_job_ids`，不计入候选人列表
4. 如果 `meta.json.recorded_job_ids` 为空数组，Step 1 等价于全部通过
5. 如果 `meta.json.exclude_keywords` 为空数组，Step 2 等价于全部通过
6. 每轮筛选结束后，将通过的岗位的 `security_id` 收集起来，传递给步骤 7 由其统一追加到 `meta.json.recorded_job_ids`，防止下一轮重复推荐。筛选环节本身不修改 meta.json
7. `job-excluded.csv` 中的岗位在 Step 1.5 被排除，不会进入候选列表。`recorded_job_ids` 已包含这些 ID，此步骤作为额外安全网
