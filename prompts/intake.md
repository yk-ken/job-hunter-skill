# 新用户引导 — 求职画像录入

本文件定义新用户首次运行 `/job-hunter` 时的完整引导流程。

执行顺序：环境检测 → 分步问答 → 生成画像文件 → 初始化运行数据。

---

## 第一步：环境检测

在开始问答之前，必须先确认运行环境就绪。逐项执行以下检测，并将结果展示给用户。

### 检测项 1：Node.js 版本

执行命令：

```bash
node --version
```

- 如果输出 >= v20.x.x：通过
- 如果输出 < v20.x.x：未通过，提示：

  ```
  Node.js 版本不满足要求（需要 >= 20，当前为 {version}）。
  请前往 https://nodejs.org/ 下载并安装 Node.js 20 或更高版本。
  ```

- 如果命令不存在：未通过，提示：

  ```
  未检测到 Node.js。
  请前往 https://nodejs.org/ 下载并安装 Node.js 20 或更高版本。
  ```

### 检测项 2：opencli 已安装

执行命令：

```bash
opencli --version
```

- 如果输出正常（有版本号）：通过
- 如果命令不存在：未通过，询问用户是否愿意安装：

  使用 AskUserQuestion 询问：
  - 问题："未检测到 opencli（Boss 直聘等网站命令行工具）。是否需要我帮你安装？"
  - 选项：["是，请帮我安装", "我自己安装，稍后继续"]
  - 如果用户选择安装：执行 `npm install -g opencli`，然后重新检测
  - 如果用户选择自行安装：给出安装指引：

    ```
    安装方式（任选其一）：
      npm install -g opencli
    安装完成后请重新运行 /job-hunter
    ```

### 检测项 3：Browser Bridge 连接

执行命令：

```bash
opencli doctor
```

- 如果显示 Browser Bridge 已连接：通过
- 如果显示 Browser Bridge 未连接：未通过，给出指引：

  ```
  Browser Bridge Chrome 扩展未连接。请按以下步骤操作：

  1. 打开 Chrome 浏览器
  2. 访问 Chrome Web Store，搜索 "Browser Bridge" 扩展并安装
  3. 安装后点击扩展图标，确保服务已启动
  4. 回到这里重新运行 /job-hunter

  注意：Browser Bridge 需要与 opencli 配合使用，请确保两者版本兼容。
  ```

### 检测项 4：Boss 直聘登录状态

执行命令：

```bash
opencli boss search "测试" --limit 1 -f json
```

- 如果返回正常结果（有数据或提示无结果但不报错）：通过
- 如果报错或返回登录相关信息：未通过，给出指引：

  ```
  Boss 直聘未登录或登录已过期。请按以下步骤操作：

  1. 在 Chrome 浏览器中打开 https://www.zhipin.com
  2. 完成登录（扫码或账号密码）
  3. 确认能看到首页推荐职位
  4. 回到这里重新运行 /job-hunter
  ```

### 检测结果展示

将四项检测结果以表格形式展示给用户：

```
环境检测结果：

  [PASS] Node.js .............. {version}
  [PASS] opencli .............. {version}
  [PASS] Browser Bridge ....... 已连接
  [PASS] Boss 直聘登录 ........ 已登录

所有环境就绪，开始建立求职画像！
```

如果有未通过项：

```
环境检测未通过，以下问题需要解决：

  [PASS] Node.js .............. {version}
  [FAIL] opencli .............. 未安装
  [----] Browser Bridge ....... 跳过（依赖 opencli）
  [----] Boss 直聘登录 ........ 跳过（依赖 Browser Bridge）

请解决以上问题后重新运行 /job-hunter。
```

注意：如果前置依赖缺失（如 opencli 未安装），后续检测项跳过并标记为 `[----]`。

**全部通过后，进入分步问答。**

---

## 第二步：分步问答

使用 AskUserQuestion 逐组收集求职画像信息。每组最多 4 个问题，给出清晰的提示和默认值。

### 第 1 组：基本信息

使用一次 AskUserQuestion 调用，包含以下 4 个问题：

- 问题 1："你想在哪个城市工作？"
  - 类型：自由输入（text）
  - 默认值：无
  - 说明：输入城市名称，如"广州"、"深圳"

- 问题 2："优先考虑哪些区域？（可多选）"
  - 类型：自由输入（text）
  - 默认值：无
  - 说明：输入区域名称，多个用逗号分隔，如"天河区, 海珠区"。不确定可留空。

- 问题 3："有哪些区域你明确不想去？（排除区域）"
  - 类型：自由输入（text）
  - 默认值：无
  - 说明：输入区域名称，多个用逗号分隔。没有可留空。

- 问题 4："你的工作年限是多少？"
  - 类型：自由输入（text）
  - 默认值：无
  - 说明：输入数字，如"5"表示 5 年经验

### 第 2 组：薪资与岗位方向

使用一次 AskUserQuestion 调用，包含以下问题：

- 问题 1："你的期望薪资范围？（最低 / 期望 / 最高，单位：K）"
  - 类型：自由输入（text）
  - 默认值：无
  - 说明：格式示例："15/18/22"，表示最低 15K、期望 18K、最高 22K

- 问题 2："你倾向哪些岗位方向？（可多选）"
  - 类型：自由输入（text）
  - 默认值：无
  - 说明：可选值示例：全栈开发、后端开发、前端开发、AI 应用开发、数据开发、DevOps、移动开发。也可自行输入其他方向，多个用逗号分隔。

### 第 3 组：技术栈与休息制度

使用一次 AskUserQuestion 调用，包含以下 2 个问题：

- 问题 1："你掌握的技术栈有哪些？"
  - 类型：自由输入（text）
  - 默认值：无
  - 说明：自由列出你熟悉的语言、框架、工具。格式示例："Java, Go, Python, Vue, MySQL, Redis, Docker"。按熟悉程度排列。

- 问题 2："你对休息制度有什么要求？"
  - 类型：自由输入（text）
  - 默认值：无
  - 说明：可选值示例：双休、单双休（大小周）、不限。也可写"最低接受单双休，期望双休"。

### 第 4 组：公司偏好与搜索关键词

使用一次 AskUserQuestion 调用，包含以下 3 个问题：

- 问题 1："你偏好什么类型的公司？"
  - 类型：自由输入（text）
  - 默认值：无
  - 说明：可选值示例：外企、国企、大厂、中大型企业、创业公司。可排序，如"外企 > 大厂 > 创业公司"。可不填表示不限。

- 问题 2："对公司规模有要求吗？"
  - 类型：自由输入（text）
  - 默认值：无
  - 说明：可选值示例：100人以上、50人以上、不限。也可写"最低50人（以开发为主体）"。可不填表示不限。

- 问题 3："有没有自定义的搜索关键词建议？"
  - 类型：自由输入（text）
  - 默认值：无
  - 说明：这些关键词会直接用于 Boss 直聘搜索。格式示例："全栈开发 广州, AI开发 广州"。如果留空，系统会根据你的岗位方向和城市自动生成。

## 第三步：生成画像文件

问答完成后，将收集的信息按以下格式写入 `data/job-profile.md`。

如果用户在第 4 组问题中没有提供搜索关键词，则根据岗位方向 + 城市自动生成默认关键词。生成规则：

- 将每个岗位方向与城市名组合，如"全栈开发 广州"、"AI应用开发 广州"
- 如果用户提供了技术栈中的核心语言，额外生成一组，如"Go 广州"、"Python 广州"
- 最多生成 4 组关键词

写入文件路径：`data/job-profile.md`

文件内容格式：

```markdown
# 求职画像

## 基本信息
- 城市：{城市}
- 优先区域：{优先区域，逗号分隔，无则写"（无）"}
- 排除区域：{排除区域，逗号分隔，无则写"（无）"}
- 工作年限：{年限}年

## 薪资期望
- 最低：{最低}K
- 期望：{期望}K
- 最高：{最高}K

## 岗位方向
{每行一个方向，用 "- " 开头}

## 技术栈
{按用户输入分组，每行一组，用 "- " 开头。如用户输入 "Java, Go, Python, Vue, MySQL"，
则按类别整理：
- Java, Go, Python
- Vue
- MySQL, Redis
- Docker
如果用户未明确分类，则直接按逗号分隔列出。}

## 休息制度
- 最低：{从用户回答中提取最低接受值}
- 期望：{从用户回答中提取期望值}

## 公司偏好
- 类型优先级：{用户偏好排序，未填则写"不限"}
- 最低规模：{用户要求，未填则写"不限"}

## 搜索关键词
{每行一组关键词，用 "- " 开头。如：
- 全栈开发 广州
- AI开发 广州
- Go后端 广州
- Python 广州
}
```

写入完成后，向用户确认：

```
求职画像已生成 → data/job-profile.md
你可以随时直接编辑这个文件来更新画像。
```

## 第四步：初始化运行数据

### 4.1 创建候选岗位文件

写入文件路径：`data/job-candidates.md`

文件内容：

```markdown
# 候选岗位列表

<!-- 由 Job Hunter Skill 自动管理，请勿手动修改编号和结构 -->
<!-- 新岗位会按日期分组追加到文件顶部 -->
```

### 4.2 创建运行元数据文件

写入文件路径：`data/meta.json`

文件内容（将占位符替换为实际值）：

```json
{
  "created_at": "{当前ISO时间，如 2026-04-05T10:00:00Z}",
  "updated_at": "{当前ISO时间}",
  "last_run_time": null,
  "cron_job_id": null,
  "run_count": 0,
  "total_candidates": 0,
  "next_candidate_number": 1,
  "search_interval_minutes": 30,
  "recorded_job_ids": [],
  "exclude_keywords": [],
  "search_keywords": [
    "{从画像的搜索关键词部分逐条写入}"
  ]
}
```

注意事项：
- `created_at` 和 `updated_at` 使用当前时间的 ISO 8601 格式
- `last_run_time` 初始为 null
- `cron_job_id` 初始为 null（创建定时任务时更新）
- `search_interval_minutes` 默认为 30
- `recorded_job_ids` 初始为空数组
- `exclude_keywords` 初始为空数组
- `search_keywords` 从画像文件的"搜索关键词"部分提取

写入完成后，向用户确认：

```
运行数据已初始化：
  - data/job-candidates.md — 候选岗位记录文件
  - data/meta.json — 运行元数据

接下来将创建定时搜索任务。
```

初始化完成后，流程继续进入环节 4（创建定时任务），由 SKILL.md 主流程接管。
